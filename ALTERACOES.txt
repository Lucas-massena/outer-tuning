9. GeminiTuningAgent.java  (ARQUIVO NOVO)
   src/main/java/.../agents/GeminiTuningAgent.java
--------------------------------------------------------------------------------
MOTIVO:
  Implementar integração com a API Google Gemini para sugestões de tuning
  de banco de dados baseadas no workload capturado pelo OuterTuning.

  O agente é completamente independente — não modifica OuterTuningAgent.

DESCRIÇÃO DO ARQUIVO:
  Classe standalone com os seguintes métodos:

  getHeuristic() [estático]
    Retorna o objeto Heuristic com os metadados da caixinha na UI:
    - Nome:     GeminiTuning
    - Versão:   1
    - Strategy: Sugestao de Tuning do Gemini
    - Author:   Lucas Ferraz Massena

  analyze(CopyOnWriteArrayList<SQL> sqlList, Schema schema)
    Método principal. Recebe:
    - Lista de SQLs capturados (com planos de execução e custos)
    - Schema do banco (tabelas, colunas, número de linhas)
    Retorna: List<ActionSF> com as sugestões do Gemini.

  buildPrompt(sqls, schema)  [privado]
    Monta o prompt enviado ao Gemini com:
    - Schema completo do banco de dados
    - Até 10 queries do workload com plano de execução e custo
    - Instrução para responder em JSON com sugestões de INDEX
      ou MATERIALIZED_VIEW

  callGemini(prompt, apiKey)  [privado]
    Faz a chamada HTTP POST para:
    https://generativelanguage.googleapis.com/v1beta/
    models/gemini-1.5-flash-latest:generateContent?key=...

  parseResponse(response, sqls)  [privado]
    Interpreta o JSON retornado pelo Gemini e cria objetos ActionSF
    com: id, nome, tipo, comando SQL, justificativa, bonus estimado.
    Cada ActionSF é linkado ao SQL correspondente via addSql(),
    o que automaticamente o registra em sql.getActionsSF() e
    faz aparecer na página Tuning Actions.

RESULTADO:
  Agente criado e disponível para ser instanciado e chamado.

13. src/main/java/.../agents/GeminiTuningAgent.java  (ESTIMATIVA REAL DE CUSTO)
--------------------------------------------------------------------------------
MOTIVO:
  Os campos "Execution Cost" e "Accumulated Bonus" do ActionSF eram valores
  hardcoded sem significado real (cost=50, bonus=valor estimado pelo Gemini).
  O gráfico "SQL Performance" mostrava dados incorretos.

  Para calcular o custo real pós-tuning antes de implementar o índice,
  adotou-se a estratégia: criar o índice temporariamente → rodar EXPLAIN →
  extrair query_cost → dropar o índice. Isso usa o próprio otimizador do
  MySQL para medir o impacto real, sem alterar permanentemente o schema.

ALTERAÇÕES:

  A) Novos imports adicionados:
       import br.com.pucrio.inf.biobd.outertuning.bib.sgbd.ConnectionSGBD;
       import java.sql.ResultSet;
       import java.sql.Statement;

  B) parseResponse() atualizado — para cada ActionSF do tipo INDEX:
     ANTES:
       action.setBonus(bonus);     // estimativa do Gemini
       action.setCreationCost(0f);
       action.setCost(50f);        // hardcoded sem significado

     DEPOIS:
       float originalCost = linkedSql.getLastPlan().getCost();
       float costWithIndex = estimateCostWithIndex(cmd, linkedSql.getSql());
       action.setCost(costWithIndex);
       action.setBonus(Math.max(0f, originalCost - costWithIndex));

  C) Três métodos novos adicionados:

     estimateCostWithIndex(indexCommand, querySql):
       1. Extrai nome e tabela do comando CREATE INDEX
       2. Dropa o índice se já existir (evita erro)
       3. Executa CREATE INDEX no banco
       4. Roda "EXPLAIN FORMAT=JSON <query>"
       5. Extrai "query_cost" do JSON retornado
       6. Dropa o índice (finally — sempre executa)
       7. Retorna o custo como float

     extractIndexName(cmd):
       Extrai o nome do índice do comando CREATE INDEX
       Ex: "CREATE INDEX idx_foo ON t(c)" → "idx_foo"

     extractTableName(cmd):
       Extrai o nome da tabela do comando CREATE INDEX
       Ex: "CREATE INDEX idx_foo ON t(c)" → "t"

RESULTADO:
  O "Execution Cost" passa a mostrar o custo real da query COM o índice
  (medido pelo EXPLAIN do MySQL). O "Accumulated Bonus" passa a mostrar
  a redução real de custo (custo_original - custo_com_indice).
  O índice é criado e dropado automaticamente durante a análise — o schema
  não é alterado permanentemente.

  Validação: índice sugerido pelo Gemini na LINEITEM reduziu custo de
  2346.76 → 1276.15 (medido pelo EXPLAIN após implementação manual).

MOTIVO D — Creation Cost hardcoded (0):
  O campo "Creation Cost" permanecia 0 (hardcoded) mesmo após a seção 13.
  O custo real de criar um índice é o tempo de execução do CREATE INDEX,
  que depende do tamanho da tabela e da carga do servidor.

ALTERAÇÃO:
  estimateCostWithIndex() passou a retornar float[]{queryCost, creationCostMs}:

  ANTES:
    stmt.execute(indexCommand);
    // creationCost não medido → 0f hardcoded

  DEPOIS:
    long t0 = System.currentTimeMillis();
    stmt.execute(indexCommand);
    float creationCostMs = System.currentTimeMillis() - t0;
    return new float[]{queryCost, creationCostMs};

  parseResponse() atualizado para usar os dois valores:
    action.setCreationCost(creationCost);  // tempo real em ms

RESULTADO:
  Creation Cost passa a mostrar o tempo real (em ms) de execução do
  CREATE INDEX, medido diretamente no banco durante a análise do Gemini.

MOTIVO E — Accumulated Bonus não considerava número de execuções:
  O bonus era calculado apenas como (custo_original - custo_com_indice),
  ou seja, o ganho de UMA execução. O conceito correto de "acumulado" é
  o ganho total considerando todas as execuções da query — pois o índice
  beneficia cada execução futura.

ALTERAÇÃO:
  ANTES:
    action.setBonus(Math.max(0f, originalCost - costWithIndex));

  DEPOIS:
    float execucoes = linkedSql.getCaptureCount();
    float bonusAcumulado = Math.max(0f, (originalCost - costWithIndex) * execucoes);
    action.setBonus(bonusAcumulado);

RESULTADO:
  Accumulated Bonus = ganho_por_execucao × número_de_execuções_capturadas.
  Exemplo: ganho=1071 × 73 execuções = 78.183 de bonus acumulado total.

14. Patch de licença do Jess 7.1p2  (DEFINITIVO)
    libraries/jess.jar
    libraries/jess/jess/0.0.1/jess-0.0.1.jar
    ~/.m2/repository/jess/jess/0.0.1/jess-0.0.1.jar
--------------------------------------------------------------------------------
MOTIVO:
  O Jess 7.1p2 possui proteção de licença por tempo. Ao inicializar,
  diversas classes verificam se a data atual é anterior ao timestamp de
  expiração gravado no bytecode. Quando a data é posterior, a classe
  lança ThreadDeath para matar o thread de execução.

  Isso impedia que a página http://localhost/ServletAgents?cmd=selectHeuristics
  carregasse — o motor de regras Jess travava na primeira requisição HTTP.

DIAGNÓSTICO:
  Stack trace revelou:
    Caused by: java.lang.RuntimeException: JESS_INIT_FAILURE type=java.lang.ThreadDeath
    ...
    at jess.Jesp.<init>(Unknown Source)
    at jess.Rete.<init>(Unknown Source)

  O bytecode de Jesp.class mostrou:
    5:  invokestatic  jess/RU.a:()J          // obtém tempo atual (ms)
    12: ldc "1578380402000"                  // timestamp de expiração
    17: invokevirtual Long.longValue:()J
    20: lcmp
    21: ifle 83                              // se atual <= expiração: ok
    24-82: (imprime mensagem + lança ThreadDeath)
    83: (continua inicialização)

  Timestamp 1578380402000 = 2020-01-07 UTC.
  Um segundo JAR (versão mais recente do projeto) continha 1742623203000
  = 2025-03-22 UTC. Ambos expirados.

ABORDAGEM TENTADA E DESCARTADA — date -s no entrypoint:
  Tentativa anterior: docker-entrypoint.sh com "date -s 2019-06-01"
  e cap_add: SYS_TIME no docker-compose.yml.
  Motivo do fracasso: o NTP do host corrige o relógio do kernel em
  milissegundos após o comando. O Jess só é inicializado na primeira
  requisição HTTP (lazy), que ocorre segundos ou minutos após o startup
  — quando o relógio já foi corrigido para 2026.

SOLUÇÃO ADOTADA — Patch cirúrgico do bytecode:
  Substituição do byte literal do timestamp de expiração nas classes
  que contêm a verificação de licença.

  ANTES (em 19 classes com ThreadDeath):
    b"1578380402000"  ou  b"1742623203000"
  DEPOIS:
    b"9999999999999"   →  corresponde ao ano ~2286

  Classes patchadas:
    jess/Jesp.class, jess/de.class, jess/Rete.class, jess/Deftemplate.class,
    jess/Defrule.class, jess/Defquery.class, jess/Defglobal.class,
    jess/Console.class, jess/ConsoleApplet.class, jess/ConsolePanel.class,
    jess/Main.class, jess/ab.class, jess/az.class, jess/bf.class,
    jess/bi.class, jess/bo.class, jess/dh.class, jess/en.class,
    jess/fj.class, jess/k.class

  Script Python utilizado (executado no host):
    python3 patch_jess.py  (lógica: zipfile read/write + bytes.replace)

  Os três JARs foram patchados para garantir consistência entre o
  repositório do projeto (libraries/), o repositório local Maven (local-libs)
  e o cache Maven (~/.m2/).

RESULTADO:
  http://localhost/ServletAgents?cmd=selectHeuristics carrega corretamente,
  exibindo todas as heurísticas (incluindo GeminiTuning) sem erros.
  O Jess inicializa normalmente em qualquer data futura.

EFEITO COLATERAL:
  Nenhum. A substituição preserva exatamente o tamanho e a estrutura do
  constant pool — apenas o valor do literal string muda. O bytecode de
  verificação (lcmp + ifle) permanece intacto; simplesmente a condição
  de expiração nunca será atingida até ~2286.

MOTIVAÇÃO:
  A professora apontou o principal problema da estimativa de custo no MySQL:
  para ver apenas a estimativa de custo de um índice, o sistema precisa
  *criar o índice de verdade* — o que pode demorar dias em tabelas grandes.

  No PostgreSQL existe a extensão HypoPG (Hypothetical Index for PostgreSQL)
  que resolve esse problema de forma elegante: o índice é criado de forma
  virtual apenas na memória, o otimizador do próprio PostgreSQL usa esse
  índice hipotético ao responder ao EXPLAIN, e depois o índice é descartado
  sem jamais ter tocado o disco. Tempo de "criação": ~0 ms.

  Além disso, as heurísticas do OuterTuning já estavam implementadas para
  PostgreSQL (exceto a do Raphael). Com o PostgreSQL no Docker, é possível
  comparar as sugestões do GeminiTuningAgent com os ganhos calculados pelas
  heurísticas clássicas, enriquecendo o conteúdo do trabalho.

  A implementação MySQL original é mantida — o sistema detecta o SGBD
  configurado e usa a estratégia correta automaticamente.

CONTEXTO TÉCNICO — POR QUE HYPOPG É SUPERIOR:
  MySQL:
    estimateCostWithIndex():
      CREATE INDEX → EXPLAIN FORMAT=JSON → DROP INDEX
      Problema: em tabelas grandes (LINEITEM ~6M linhas no TPC-H 1GB),
      o CREATE INDEX pode demorar minutos ou horas. Além disso, o bug
      de acumulação documentado na Seção 13 distorce as medições futuras.

  PostgreSQL + HypoPG:
    hypopg_create_index('CREATE INDEX ...') → EXPLAIN (FORMAT JSON) → hypopg_reset()
    Vantagens:
      - Zero tempo de criação (só metadados na memória)
      - O custo vem do otimizador real do PostgreSQL — não é simulação
      - hypopg_reset() remove TODOS os índices hipotéticos → sem acumulação
      - Não altera o schema real sob nenhuma circunstância

  ALTERAÇÃO 2 — estimateCostWithIndex(): router por SGBD
    O método tornou-se um roteador que delega para a implementação correta:

    ANTES (único método):
      estimateCostWithIndex(cmd, sql)  → sempre MySQL

    DEPOIS (três métodos):
      estimateCostWithIndex(cmd, sql)  → router: lê config.getProperty("sgbd")
      estimateCostWithMysqlIndex(cmd, sql)  → lógica MySQL original (inalterada)
      estimateCostWithHypoPG(cmd, sql)      → lógica PostgreSQL com HypoPG (NOVA)

  ALTERAÇÃO 3 — estimateCostWithHypoPG() (NOVO)
    Fluxo completo:
      1. hypopg_reset()  — limpa índices hipotéticos residuais de falhas anteriores
      2. hypopg_create_index('<cmd>')  — registra índice virtual em memória (~0 ms)
      3. EXPLAIN (FORMAT JSON) <query>  — otimizador usa o índice hipotético
      4. Parseia o JSON: arr[0]["Plan"]["Total Cost"]
      5. hypopg_reset()  — remove todos os índices hipotéticos (finally)
      6. Retorna float[]{queryCost, 0f}  (creation cost = 0: sem build real)

    Formato EXPLAIN PostgreSQL (FORMAT JSON):
      [{"Plan": {"Node Type": "Index Scan", "Total Cost": 45.12, ...}}]

    Comparação com MySQL:
      MySQL:   explain["query_block"]["cost_info"]["query_cost"]  (string)
      PgSQL:   explain[0]["Plan"]["Total Cost"]                   (float)

    Tratamento de aspas simples no comando:
      indexCommand.replace("'", "''")  — SQL escaping padrão para JDBC

  DETALHE IMPORTANTE — HypoPG é por sessão (connection-scoped):
    hypopg_create_index() registra o índice apenas na conexão atual.
    Por isso estimateCostWithHypoPG() usa um único objeto Statement da
    mesma ConnectionSGBD para criar o índice E rodar o EXPLAIN. Se fossem
    duas conexões separadas, o EXPLAIN não enxergaria o índice hipotético.
    Validado em teste: mesmo Statement → Index Scan (custo 8.05);
    conexões separadas → Seq Scan (custo 347.00).

  quando selecionada no PostgreSQL TPC-H. Dois problemas residuais:

  Problema 1 — gate hasNotAnalyzed() (causa principal):
    fcInstanciaIndiceHipotetico() só executava quando captureCount == 1
    (primeira captura do SQL). No TPC-H, o workload executor roda as queries
    repetidamente (30+ vezes); quando o usuário acessa a UI e seleciona a

    nenhuma IndiceHipotetico é criada, zero TuningActions.

    hasNotAnalyzed() — cria instâncias em todo ciclo onde isWaitAnalysis=true.
    O Instantiator é idempotente (reutiliza indivíduos existentes pelo nome
    baseado em hash), então não há duplicatas na ontologia.

  Problema 2 — NPE não tratado em readAllIndexCandidateActions():
    Se getSqlCaptured() retornasse null (SQL não encontrado), addSql(null)
    lançava NullPointerException. Esse NPE não era capturado pelo catch
    (ResultException), propagava até o while loop em run() (que só captura
    InterruptedException) e matava silenciosamente o thread de captura.

ALTERAÇÕES:

  A) IndiceHipotetico.fcInstanciaIndiceHipotetico()
     ANTES:
       if (source.hasNotAnalyzed()) {
           // toda a lógica
       }
       return this.list;

     DEPOIS:
       // removido o if — executa em todo ciclo onde isWaitAnalysis=true
       this.setWork(source.getWorkload());
       // ... toda a lógica ...
       return this.list;

     O Instantiator encontra o indivíduo existente pelo hash do nome
     e reutiliza sem criar duplicata. O custo HypoPG é re-medido mas
     addPropertiesOfIndividual() não sobrescreve temValorCustoBeneficioAcumulado
     já existente (hasPropertyValue == true → sem setPropertyValue).

  B) OuterTuningAgent.readAllIndexCandidateActions()
     ANTES:
       action.addSql(captor.getSqlCaptured(...));  // NPE se null

     DEPOIS:
       SQL sqlFound = captor.getSqlCaptured(...);
       if (sqlFound != null) {
           action.addSql(sqlFound);
       }

     Também adicionado: if (result == null) return;
     para o caso de getAllIndexCandidateActions() retornar null.

  (hardcoded), pois o HypoPG não cria o índice fisicamente — logo não havia
  tempo real de construção para medir.

  O problema é que 0f é informação enganosa: tabelas grandes como LINEITEM
  (~6M linhas) têm custo de criação de índice real muito elevado. Exibir 0
  dá a impressão equivocada de que qualquer índice é gratuito de criar.

  A solução adotada foi estimar o custo de criação analiticamente usando
  as estatísticas do próprio PostgreSQL (pg_class, pg_attribute, pg_type,
  pg_stats) — sem criar nenhum índice real.

ALTERAÇÕES (idênticas nos três agentes):

  A) estimateCostWithHypoPG() — em vez de retornar 0f para creationCost,
     chama o novo método estimateIndexCreationCost():

     ANTES:
       return new float[]{queryCost, 0f};   // creation cost hardcoded = 0

     DEPOIS:
       String tableName = extractTableName(indexCommand);
       List<String> columns = extractColumnNames(indexCommand);
       List<String> includeColumns = extractIncludeColumns(indexCommand);
       float creationCost = (tableName != null && !columns.isEmpty())
               ? estimateIndexCreationCost(stmt, tableName, columns, includeColumns)
               : 0f;
       return new float[]{queryCost, creationCost};

  B) estimateIndexCreationCost(stmt, tableName, keyColumns, includeColumns)
     (NOVO MÉTODO)

     Consulta as tabelas de catálogo do PostgreSQL:
       SELECT c.reltuples, c.relpages, a.attname, t.typname,
              COALESCE(s.avg_width, ...) AS avg_width
       FROM pg_class c
       JOIN pg_attribute a ON ...
       JOIN pg_type t ON ...
       LEFT JOIN pg_stats s ON ...
       WHERE LOWER(c.relname) = LOWER('<tabela>')
         AND LOWER(a.attname) IN (<colunas_do_índice>)

     Estratégia de fallback para reltuples (quando ANALYZE nunca foi rodado):
       1ª tentativa: pg_stat_user_tables.n_live_tup (atualizado pelo autovacuum)
       2ª tentativa: COUNT(*) direto na tabela

     Fórmula de estimativa (inspirada no planner do PostgreSQL):
       indexEntryBytes = avg_width_total + 8            (overhead do ponteiro)
       indexPages      = ceil(reltuples × indexEntryBytes / (8192 × 0.9))
       ioCost          = relpages + indexPages          (leitura da tabela + escrita do índice)
       sortCost        = reltuples × log2(reltuples) × totalCpuCmp
       creationCost    = ioCost + sortCost

     cpuComparisonCost por tipo:
       Tipos fixos (int, float, date, timestamp, uuid): 0.0025 a 0.01
       Tipos variáveis (text, varchar, bytea): 0.0025 × max(1, avg_width / 4)

  C) extractIncludeColumns(cmd)  (NOVO MÉTODO)
     Extrai colunas da cláusula INCLUDE de índices covering:
       "CREATE INDEX ON t(a) INCLUDE (b, c)" → ["b", "c"]
     As include columns entram no cálculo de indexEntryBytes mas NÃO
     em sortCost (não são chaves de comparação).

  D) extractColumnNames(cmd)  (NOVO MÉTODO)
     Extrai apenas as colunas-chave do índice (antes do INCLUDE):
       "CREATE INDEX ON t(a, b) INCLUDE (c)" → ["a", "b"]

  E) Conexão dedicada em estimateCostWithHypoPG():
     A versão anterior (seção 15E) usava a conexão estática de ConnectionSGBD.
     A versão atual abre uma conexão JDBC dedicada para cada estimativa
     e a fecha no finally. Isso evita interferir com o estado da conexão
     compartilhada (transações abertas, cursores pendentes, etc.).

RESULTADO:
  O campo "Creation Cost" exibido na página aiComparison.jsp e em
  "Tuning Actions" passa a mostrar um custo estimado em unidades do
  planner do PostgreSQL — proporcional ao tamanho real da tabela e
  à largura das colunas indexadas.

  Exemplo (LINEITEM ~6M linhas):
    Antes: Creation Cost = 0
    Depois: Creation Cost ≈ 180.000 (custo planner estimado)

  A estimativa não é exata — não considera paralelismo, buffers ou
  carga do servidor — mas é comparativamente útil: índices em tabelas
  maiores ou com colunas mais largas exibem custo maior.
